--- KEY TECHNICAL DETAILS (must be implemented exactly)

1) ARCore pose → COLMAP conversion (exact math)
   Background (ARCore):
     - `Pose` from ARCore gives a rigid transform from local (camera) → world: X_world = R_cw * X_cam + t_cw.
     - For rendering AR content use `getDisplayOrientedPose()` which is adjusted for display rotation (+X right, +Y up, -Z camera viewing direction). Use this for easier UI mapping.
   Required (COLMAP expects world→camera):
     - COLMAP uses a quaternion (QW QX QY QZ) + translation (TX TY TZ) that express the projection from **world → camera**:
       X_cam = R_wc * X_world + T_wc
     - Therefore compute:
         R_cw = rotation matrix from ARCore Pose (camera→world)
         t_cw = translation vector from ARCore Pose (camera→world)
         R_wc = transpose(R_cw)          # world -> camera rotation
         T_wc = - R_wc * t_cw            # world -> camera translation
     - Convert R_wc to quaternion q = (q_w, q_x, q_y, q_z) using the Hamilton (w-first) convention used by COLMAP (and Eigen). Ensure quaternion sign convention matches COLMAP (w first).
     - The final COLMAP image line uses: IMAGE_ID QW QX QY QZ TX TY TZ CAMERA_ID IMAGE_NAME
   Pseudocode (Kotlin style):
     // arPose: com.google.ar.core.Pose
     val r_cw = arPose.rotationMatrix()        // 3x3
     val t_cw = arPose.translation()           // 3-vector
     val r_wc = transpose(r_cw)
     val T_wc = - multiply(r_wc, t_cw)         // 3-vector
     val q = rotationMatrixToHamiltonQuaternion(r_wc) // q.w, q.x, q.y, q.z
     // Write out: "${imageId} ${q.w} ${q.x} ${q.y} ${q.z} ${T_wc[0]} ${T_wc[1]} ${T_wc[2]} ${cameraId} ${imageName}"
   Rationale & checks:
     - Verify with a simple test: place a known world point P, compute camera reprojection with both ARCore and reconstructed COLMAP camera extrinsic and check pixel projection equality.
     - Use the COLMAP doc examples for images.txt and cameras.txt formats. (Follow COLMAP text format precisely).

   References: COLMAP format & images.txt spec; ARCore Pose docs. (Use these to validate sign and axis conventions). 

2) COLMAP cameras.txt format & recommended camera model
   - Use `PINHOLE` or `OPENCV` depending on distortion model. For mobile phone prefer `PINHOLE` if you will calibrate fx,fy,cx,cy; or `OPENCV` if you want to pass radial tangential distortion.
   - Example cameras.txt line (PINHOLE):  
       CAMERA_ID PINHOLE WIDTH HEIGHT FX FY CX CY
   - Write camera intrinsics extracted from CameraX / camera characteristics:
       fx ≈ focalLengthInPixelsX = (focalLengthMM / sensorWidthMM) * imageWidthPixels
       fy ≈ focalLengthInPixelsY (similar)
       principal point cx, cy from sensor active array or use imageWidth/2, imageHeight/2 if unavailable.
   - For robust results, compute focal length from the Android CameraCharacteristics.LENS_INFO available via Camera2 API (CameraX extension).

3) Frame selection algorithm (auto-frame selection)
   Objective: pick high-quality and well-spaced frames for COLMAP input.
   Metrics per frame (compute per captured frame before writing):
     A. Motion delta (poseDelta):
        - angleDelta = angle between current and last accepted camera orientation (degrees).
        - transDelta = Euclidean distance between current and last accepted camera position (meters).
        - Accept if angleDelta >= 5° OR transDelta >= 0.05 m.
     B. Sharpness (variance of Laplacian):
        - Compute Laplacian(var) over grayscale image patch; require var > sharpnessThreshold (e.g., 50).
     C. Depth consistency:
        - Compute mean depth in center region from depth image and depth variance across patch. Reject if depth variance too high (blurry depth) or depth outside [0.25, 1.5] m depending on object size.
     D. Exposure stability:
        - Compare image histogram mean/median to last accepted; reject if big excursion.
     E. Tracking quality:
        - ARCore.getCamera().trackingState must be TRACKING.
     F. Feature density:
        - Run ORB/FAST detector or use Android's ImageAnalysis to run a lightweight feature detector; require a minimum number of features (e.g., 200).
   Frame Score:
     score = w1*norm(pose_metric) + w2*norm(sharpness) + w3*norm(feature_density) + w4*norm(depth_consistency) + w5*tracking_flag
     where weights sum to 1; tune empirically (example: w1=0.35,w2=0.25,w3=0.2,w4=0.15,w5=0.05).
   Acceptance rule:
     - If score > threshold (e.g., 0.5) and motion spacing satisfied → accept frame and write image+metadata.
   Implementation:
     - Implement FrameProcessor.kt that receives per-frame: bitmap, depthImage (float[]), ARCore Pose, timestamp.
     - Use a small worker thread to compute Laplacian variance using OpenCV (or simple convolution).
     - Maintain lastAcceptedPose and lastAcceptedTimestamp.
     - Provide manual capture override (UI button).

4) Orbit coverage heatmap & guidance UI
   - Define the object center (anchor) upon user tap or plane estimate (tap to set anchor is recommended).
   - For each accepted frame compute azimuth angle relative to object center:
       dx = camPos.x - objCenter.x
       dz = camPos.z - objCenter.z
       azimuth = atan2(dz, dx) in degrees [−180,180]
       elevation = asin((camPos.y - objCenter.y) / distance)
   - Discretize azimuth into N bins (e.g., 36 bins = 10°). Maintain coverage count per bin (increment when frame accepted).
   - Visual UI: circular ring segmented into bins; color intensity = normalized coverage. Provide recommended next orbit: low, mid, high (elevation bands).
   - Provide heatmap thresholds: include flips (if user flips object), show missing sectors in red.
